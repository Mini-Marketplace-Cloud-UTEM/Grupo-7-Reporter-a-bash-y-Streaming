🗄️ Motor de Base de Datos

Supabase PostgreSQL — proyecto en producción. Conexión asíncrona via SQLAlchemy 2.0 + asyncpg. URL configurada como variable de entorno DATABASE_URL.

📁 Migraciones SQL

Tres archivos de migración versionados bajo migrations/, ejecutados automáticamente por Docker Compose al iniciar el servicio en desarrollo:

  • 001_initial_schema.sql — tablas de ventas y productos
  • 002_inventory_shortage.sql — log de quiebres de stock
  • 003_batch_jobs.sql — tabla de trabajos batch

📄 Headers requeridos

  • X-Correlation-Id — UUID, obligatorio

Entregable 4

Evidencia CI/CD y Deploy

GitHub Actions ejecuta lint + tests en cada push. Render.com despliega automáticamente desde main.

⚙️ Pipeline GitHub Actions (ci.yml)

🔍
Job: lint (ruff)
Ejecuta ruff check y ruff format --check sobre todo el código Python. Falla si hay errores de estilo o imports no usados.
✅
Job: test (pytest) — necesita lint
Corre la suite completa con pytest-asyncio. Genera reporte JUnit XML y cobertura XML. Umbral mínimo: 60% global, 70% código nuevo.
📦
Artefactos — JUnit + Coverage
Publica reports/junit.xml y reports/coverage.xml como artefactos. En PRs publica resumen de cobertura como comentario automático.
🚀
Deploy — Render.com (automático)
Cada push a main dispara el deploy en Render.com via integración nativa con GitHub. Build Docker incluido en el proceso.

🐳 Dockerfile

Imagen basada en python:3.11-slim. Instala dependencias desde requirements.txt y arranca con uvicorn en modo producción.

FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install -r requirements.txt COPY . . CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8070"]

🔀 Triggers del pipeline

  • Push a main o develop
  • Pull Request hacia main o develop
  • Excluye cambios solo en docs/**
  • Cancela runs anteriores (concurrency)

🧩 Docker Compose local

  • postgres — PostgreSQL 16 Alpine con healthcheck
  • migrate — Aplica 3 migraciones SQL al iniciar
  • api — Servicio FastAPI en puerto 8070

⚠️ Casos de error cubiertos

422
Headers X-Request-Id, X-Correlation-Id o X-Consumer ausentes o con formato inválido.
400
UUID inválido en cualquiera de los headers requeridos (validación Pydantic en dependency).
409
Idempotency-Key ya usada: el batch con esa clave ya existe en BD — evita doble ejecución.
503
Simulado con X-Mock-Http-Status: 503 para probar resiliencia del BFF sin condición real de fallo.
500
Error inesperado en handler. El middleware retorna ErrorResponse estándar con timestamp ISO 8601 y correlationId.
