## Hallazgos

### [SEC-001] 🟠 ALTO: Sin autenticación real en los endpoints de la API
**Categoría**: OWASP A01 — Broken Access Control  
**Archivo**: `app/api/dependencies.py:21-65`, `app/api/routes/reports.py`, `app/api/routes/batch.py`  
**CWE**: CWE-306 — Missing Authentication for Critical Function  
**Descripción**: Los endpoints usan `require_headers` / `require_headers_with_idempotency` como única "protección". Estas dependencias solo validan que `X-Request-Id` y `X-Correlation-Id` sean UUIDs y que `X-Consumer` no esté vacío. Cualquier cliente externo puede fabricar esos headers y acceder libremente a todos los reportes analíticos o lanzar un recálculo batch.  
**Impacto**: Lectura sin restricciones de métricas de ventas, productos y rendimiento de despacho. Ejecución arbitraria del proceso batch que accede a Supabase Storage con `SUPABASE_SECRET_KEY`.  
**Evidencia**:
```python
# app/api/dependencies.py:21 — el "guard" es solo validación de formato, no autenticación
async def require_headers(
    x_request_id: str = Header(..., alias="X-Request-Id"),
    x_correlation_id: str = Header(..., alias="X-Correlation-Id"),
    x_consumer: str = Header(..., alias="X-Consumer"),
) -> dict:
    for name, value in [("X-Request-Id", x_request_id), ("X-Correlation-Id", x_correlation_id)]:
        try:
            UUID(value)  # <-- valida formato, no identidad
        except ValueError as err:
            ...
    if x_consumer.strip() == "":  # <-- cualquier string no vacío pasa
        ...
```
**Remediación**: Agregar validación de API key o token compartido. La opción más liviana (sin JWT) es un header `X-API-Key` verificado contra una variable de entorno:
```python
# app/config.py — agregar:
API_KEY: str = ""

# app/api/dependencies.py — agregar guard:
from fastapi.security import APIKeyHeader

_api_key_header = APIKeyHeader(name="X-API-Key", auto_error=True)

async def require_api_key(api_key: str = Security(_api_key_header)) -> None:
    if not settings.API_KEY or api_key != settings.API_KEY:
        raise HTTPException(status_code=401, detail={"code": "UNAUTHORIZED", "message": "API key inválida"})

# Agregar como dependencia en cada router:
router = APIRouter(prefix="/reports", dependencies=[Depends(require_api_key)])
```
**Esfuerzo estimado**: 2h  
**Referencias**: [OWASP A01](https://owasp.org/Top10/A01_2021-Broken_Access_Control/), [CWE-306](https://cwe.mitre.org/data/definitions/306.html)

### [SEC-002] 🟠 ALTO: Sin rate limiting en ningún endpoint
**Categoría**: OWASP A04 — Insecure Design  
**Archivo**: `app/main.py` (ausencia de middleware), `app/api/routes/batch.py:27`  
**CWE**: CWE-770 — Allocation of Resources Without Limits  
**Descripción**: No existe ninguna forma de throttling. El endpoint `POST /reports/batch/recalculate` es particularmente costoso: lanza un proceso en segundo plano que lista y descarga todos los archivos del bucket de Supabase Storage. Sin rate limiting, un actor puede llamarlo en bucle y agotar conexiones de DB, cuota de Supabase Storage y CPU del worker.  
**Impacto**: Degradación del servicio o DoS efectivo con bajo costo para el atacante. El endpoint de streaming SSE `/reports/stream/sales` también puede agotar conexiones de DB.  
**Evidencia**:
```python
# app/main.py — no hay ningún RateLimitMiddleware
app.add_middleware(MockStatusMiddleware)
# ← aquí debería haber rate limiting
```
**Remediación**: Usar `slowapi` (wrapper de `limits` para FastAPI):
```python
# requirements.txt: slowapi>=0.1.9
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded

limiter = Limiter(key_func=get_remote_address)
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

# En el endpoint batch (el más costoso):
@router.post("/batch/recalculate", ...)
@limiter.limit("5/minute")
async def trigger_batch_recalculate(request: Request, ...):
    ...
```
**Esfuerzo estimado**: 2h  
**Referencias**: [OWASP A04](https://owasp.org/Top10/A04_2021-Insecure_Design/), [CWE-770](https://cwe.mitre.org/data/definitions/770.html)

### [SEC-003] 🟡 MEDIO: Sin .dockerignore — el .env se copia en la imagen
**Categoría**: OWASP A05 — Security Misconfiguration  
**Archivo**: `Dockerfile:14`  
**CWE**: CWE-312 — Cleartext Storage of Sensitive Information  
**Descripción**: El `Dockerfile` usa `COPY . .` sin ningún `.dockerignore`. Si el archivo `.env` (con `SUPABASE_SECRET_KEY`, `DATABASE_URL`, etc.) existe en el directorio de build (lo que ocurre en desarrollo local), queda embebido en la imagen Docker. Cualquier persona con acceso a la imagen o a un registry puede extraer esas credenciales.  
**Impacto**: Exposición de credenciales de producción si la imagen se publica en un registry compartido o si se hace `docker save`.  
**Evidencia**:
```dockerfile
# Dockerfile:14
COPY . .   # ← copia .env, .git/, tests/, docs/ sin filtro
```
**Remediación**: Crear `.dockerignore` en la raíz:
```
.env
.env.local
.env.production
.git/
.venv/
__pycache__/
*.pyc
.pytest_cache/
.coverage
tests/
docs/
graphify-out/
reports/
*.log
```
**Esfuerzo estimado**: 30 min  
**Referencias**: [Docker .dockerignore](https://docs.docker.com/build/concepts/context/#dockerignore-files), [CWE-312](https://cwe.mitre.org/data/definitions/312.html)

### [SEC-004] 🟡 MEDIO: Dockerfile ejecuta el proceso como root
**Categoría**: OWASP A05 — Security Misconfiguration  
**Archivo**: `Dockerfile`  
**CWE**: CWE-250 — Execution with Unnecessary Privileges  
**Descripción**: El `Dockerfile` no define una directiva `USER`. Por defecto, `uvicorn` corre como `root` dentro del contenedor. Un escape del contenedor o una vulnerabilidad en las dependencias daría acceso root.  
**Evidencia**:
```dockerfile
FROM python:3.11-slim AS base
WORKDIR /app
# ← sin USER, corre como root
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8070"]
```
**Remediación**:
```dockerfile
FROM base AS runtime
COPY --from=builder /install /usr/local
COPY . .
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
USER appuser
EXPOSE 8070
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8070"]
```
**Esfuerzo estimado**: 30 min  
**Referencias**: [Docker security best practices](https://docs.docker.com/build/building/best-practices/#user), [CWE-250](https://cwe.mitre.org/data/definitions/250.html)

### [SEC-005] 🟡 MEDIO: Endpoint SSE /reports/stream/sales sin timeout ni límite de conexiones
**Categoría**: OWASP A04 — Insecure Design  
**Archivo**: `app/api/routes/reports.py:47-58`  
**CWE**: CWE-400 — Uncontrolled Resource Consumption  
**Descripción**: El generador SSE es un `while True` que solo termina cuando el cliente desconecta. Cada conexión SSE abierta mantiene activa una sesión de SQLAlchemy durante toda su vida. Sin límite de conexiones simultáneas ni timeout, un atacante puede abrir miles de conexiones SSE y agotar el pool de conexiones de la base de datos.  
**Evidencia**:
```python
# app/api/routes/reports.py:51
async def event_generator():
    while True:                          # ← sin max_iterations ni timeout
        if await request.is_disconnected():
            break
        summary = await analytics_service.get_sales_report(db, ...)  # sesión DB abierta
        yield f"data: {json.dumps(summary.model_dump())}\n\n"
        await asyncio.sleep(10)
```
**Remediación**: Añadir un límite máximo de emisiones (ej. 360 = 1 hora):
```python
async def event_generator():
    max_events = 360  # ponytail: 1h máximo, ajustar si el BFF lo necesita mayor
    for _ in range(max_events):
        if await request.is_disconnected():
            break
        summary = await analytics_service.get_sales_report(db, None, None, use_mocks=use_mocks)
        yield f"data: {json.dumps(summary.model_dump())}\n\n"
        await asyncio.sleep(10)
```
**Esfuerzo estimado**: 30 min  
**Referencias**: [CWE-400](https://cwe.mitre.org/data/definitions/400.html)

### [SEC-006] 🔵 BAJO: Credenciales de desarrollo débiles en docker-compose y .env.example
**Categoría**: OWASP A05 — Security Misconfiguration  
**Archivo**: `.env.example:14`, `docker-compose.yml:7-9`  
**CWE**: CWE-521 — Weak Password Requirements  
**Descripción**: El password por defecto para PostgreSQL en desarrollo es `reporteria`. Si alguien despliega con estos valores por defecto en un ambiente con el puerto 5432 expuesto, la base de datos es trivialmente accesible.  
**Evidencia**:
```yaml
# docker-compose.yml:7
POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-reporteria}
```
**Remediación**: Documentar en el `README` y en `.env.example` que se debe cambiar el password antes de cualquier deploy no-local. Considerar generar passwords aleatorios en el script de setup:
```bash
# scripts/setup.sh
POSTGRES_PASSWORD=$(openssl rand -hex 16)
echo "POSTGRES_PASSWORD=$POSTGRES_PASSWORD" >> .env
```
**Esfuerzo estimado**: 30 min

### [SEC-007] 🔵 BAJO: batch_service parsea JSON del storage sin validación de schema
**Categoría**: OWASP A08 — Software and Data Integrity Failures  
**Archivo**: `app/services/batch_service.py:73-91`  
**CWE**: CWE-20 — Improper Input Validation  
**Descripción**: Los archivos descargados del bucket de Supabase Storage se parsean directamente con `event.get()` sin pasar por un schema Pydantic. Si un archivo en el bucket está malformado (ej: `totalAmount` es un string no numérico, una lista, o un objeto), `Decimal(str(...))` puede fallar de formas inesperadas y marcar el job como FAILED sin procesar eventos válidos.  
**Evidencia**:
```python
# app/services/batch_service.py:82
amount = Decimal(str(payload.get("totalAmount", 0)))  # sin validación de tipo
```
**Remediación**: Reutilizar los schemas Pydantic ya definidos en `app/schemas/events.py`:
```python
# En run_batch_recalculate, reemplazar el bloque de parseo:
from app.schemas.events import OrderCreatedPayload
try:
    if event_type == "ORDER_CREATED":
        data = OrderCreatedPayload(**payload)
        amount = Decimal(str(data.totalAmount))
        await upsert_sales_from_order(db, occurred_at, amount, correlation_id)
        processed += 1
except Exception:
    logger.warning("evento_batch_invalido tipo=%s correlationId=%s", event_type, correlation_id)
    continue
```
**Esfuerzo estimado**: 1h

### [SEC-008] ℹ️ INFO: Sin CORS configurado
**Categoría**: OWASP A05 — Security Misconfiguration  
**Archivo**: `app/main.py`  
**Descripción**: No hay `CORSMiddleware`. Para una API backend-to-backend (consumida solo por el BFF en servidor), esta es la postura correcta (más restrictiva). Pero si los clientes incluyen navegadores o la Swagger UI desde otro origen, se necesita CORS explícito.  
**Acción**: Si el BFF accede desde un navegador o si se usa la Swagger UI en producción desde un dominio diferente, agregar:
```python
from fastapi.middleware.cors import CORSMiddleware
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://bff.tu-dominio.cl"],  # nunca "*" con credenciales
    allow_methods=["GET", "POST"],
    allow_headers=["X-Request-Id", "X-Correlation-Id", "X-Consumer", "Idempotency-Key"],
)
```

### [SEC-009] ℹ️ INFO: X-Consumer sin whitelist de valores válidos
**Categoría**: OWASP A01 — Broken Access Control  
**Archivo**: `app/api/dependencies.py:36-43`  
**Descripción**: `X-Consumer` solo valida que no esté vacío. Cualquier string pasa (ej: `"attacker"`, `"internal-cron"`). No hay forma de saber si el llamante es legítimo.  
**Acción**: Con la implementación de SEC-001 (API key), este hallazgo pierde relevancia. Si se implementa API key, considerar también loguear `x_consumer` en cada request para auditoría.

## Recomendaciones adicionales

1. **Agregar security headers**: FastAPI no agrega `X-Content-Type-Options`, `X-Frame-Options`, ni `Strict-Transport-Security` automáticamente. Agregar un middleware liviano o usar un reverse proxy (nginx/Render) que los inyecte.

2. **Ejecutar `pip-audit` en CI**: Agregar un step en el workflow de GitHub Actions para detectar CVEs en dependencias de forma automática.

3. **Logging con IP de cliente**: El middleware de trazabilidad en `main.py` loguea `CorrelationID` y `RequestID` pero no la IP del cliente (`request.client.host`). Útil para detectar patrones de abuso.

4. **Validar origen de mensajes Pub/Sub**: Los mensajes de Pub/Sub se parsean y procesan sin verificar su origen (no hay firma HMAC ni verificación de `source` del envelope). Si el tópico de Pub/Sub tiene permisos abiertos, mensajes maliciosos pueden llegar al worker.
