## Hallazgos

### [SEC-001] 🟠 ALTO: MockStatusMiddleware expuesto sin restricción de entorno en runtime

**Archivo**: `app/middleware/mock_status.py`, `app/main.py:48`
**Categoría**: A05 – Security Misconfiguration

**Descripción**:
El middleware `MockStatusMiddleware` permite que **cualquier cliente HTTP** cambie el status code de la respuesta enviando el header `X-MOCK-HTTP-STATUS`, siempre que `USE_MOCKS=true` en el entorno. El único control es una variable de entorno booleana que puede activarse accidentalmente en producción (por ejemplo, si un deploy arrastra un `.env` incorrecto o si una variable de CI/CD está mal configurada).

Si `USE_MOCKS=true` llega a producción, un atacante externo puede:
- Forzar respuestas `200 OK` sobre errores reales, ocultando fallos a los monitores.
- Forzar `503 Service Unavailable` en todos los endpoints, produciendo una denegación de servicio efectiva.
- Confundir al BFF consumidor (Grupo 1) con respuestas artificiales.

No existe restricción por IP, por autenticación, ni verificación en código de que `APP_ENV != "production"` antes de procesar el header.

**Evidencia**:
```python
# app/middleware/mock_status.py:33-66
async def dispatch(self, request: Request, call_next) -> Response:
    response: Response = await call_next(request)

    if not settings.USE_MOCKS:  # ← único control: variable de entorno
        return response

    raw = request.headers.get(_MOCK_HEADER)
    # ... cualquier cliente puede forzar el status code
    response.status_code = forced_status  # ← sin validación adicional
    return response
```

**Remediación**:
```python
# Opción A: bloquear el middleware en producción por código
async def dispatch(self, request: Request, call_next) -> Response:
    response: Response = await call_next(request)

    # Doble guardia: variable de entorno Y entorno no productivo
    if not settings.USE_MOCKS or settings.APP_ENV == "production":
        return response
    # ...

# Opción B: validar X-Consumer contra una lista de consumidores internos permitidos
ALLOWED_MOCK_CONSUMERS = {"group-01-test", "group-02-test"}  # solo consumidores de test
consumer = request.headers.get("X-Consumer", "")
if consumer not in ALLOWED_MOCK_CONSUMERS:
    return response
```

### [SEC-002] 🟠 ALTO: Sin autenticación ni autorización — API completamente abierta

**Archivo**: `app/api/dependencies.py`, todos los routers
**Categoría**: A01 – Broken Access Control / A07 – Identification and Authentication Failures

**Descripción**:
La API no implementa ningún mecanismo de autenticación real. La dependencia `require_headers` valida únicamente formato UUID de los headers de trazabilidad (`X-Request-Id`, `X-Correlation-Id`) y la presencia de `X-Consumer`, pero **no verifica que el llamante sea un servicio autorizado**:

- `X-Consumer` acepta cualquier string arbitrario sin validación contra una lista blanca.
- No hay API Key, JWT, mutual TLS ni ningún otro mecanismo de identidad.
- Todos los endpoints de reportería financiera (`/reports/sales`, `/reports/top-products`, `/reports/average-ticket`) son accesibles para cualquier cliente que conozca la URL y envíe tres headers UUID válidos.

Esto expone métricas de negocio sensibles (montos de ventas en CLP, productos más vendidos, patrones horarios) a actores no autorizados.

**Evidencia**:
```python
# app/api/dependencies.py:7-27
async def require_headers(
    x_request_id: str = Header(..., alias="X-Request-Id"),
    x_correlation_id: str = Header(..., alias="X-Correlation-Id"),
    x_consumer: str = Header(..., alias="X-Consumer"),  # ← cualquier valor es válido
) -> dict:
    for name, value in [("X-Request-Id", x_request_id), ("X-Correlation-Id", x_correlation_id)]:
        try:
            UUID(value)  # ← solo valida formato, no identidad
        except ValueError:
            raise HTTPException(...)
    # X-Consumer nunca se valida contra una lista de consumidores permitidos
    return {"x_request_id": ..., "x_consumer": x_consumer}
```

**Remediación**:
```python
# Opción mínima viable: whitelist de consumidores + API Key compartida
ALLOWED_CONSUMERS = {"group-01", "group-02", "group-03"}
API_KEY_HEADER = "X-API-Key"

async def require_headers(
    x_consumer: str = Header(..., alias="X-Consumer"),
    api_key: str = Header(..., alias="X-API-Key"),
    ...
) -> dict:
    if x_consumer not in ALLOWED_CONSUMERS:
        raise HTTPException(status_code=403, detail={"code": "FORBIDDEN_CONSUMER"})
    if not secrets.compare_digest(api_key, settings.INTERNAL_API_KEY):
        raise HTTPException(status_code=401, detail={"code": "UNAUTHORIZED"})
    ...
```

Para producción real, evaluar **mutual TLS** o un **API Gateway con OAuth 2.0 client credentials** entre servicios.

### [SEC-003] 🟠 ALTO: `Idempotency-Key` se valida pero nunca se implementa — riesgo de DoS por recálculo masivo

**Archivo**: `app/api/routes/batch.py:38-48`, `app/api/dependencies.py:30-60`
**Categoría**: A01 – Broken Access Control / A07 – Authentication Failures

**Descripción**:
El endpoint `POST /reports/batch/recalculate` exige un header `Idempotency-Key` (UUID) que da la apariencia de proteger contra ejecuciones duplicadas. Sin embargo, **la clave nunca se almacena ni se consulta**; el sistema acepta múltiples requests con el mismo UUID y ejecuta el batch completo en cada uno.

El proceso batch lee **todos los archivos del bucket `event-logs`** en Supabase Storage y recalcula todas las agregaciones. Sin deduplicación real, un atacante (o bug de red) puede detonar decenas de recálculos simultáneos, agotando:
- Conexiones de base de datos del pool de SQLAlchemy.
- Cuota de operaciones de Supabase Storage.
- Memoria/CPU del servidor uvicorn.

**Evidencia**:
```python
# app/api/routes/batch.py — Idempotency-Key se recibe pero se descarta
async def trigger_batch_recalculate(
    body: BatchRecalculateRequest | None = None,
    background_tasks: BackgroundTasks = BackgroundTasks(),
    _headers: dict = Depends(require_headers_with_idempotency),  # ← valida formato
    db: AsyncSession = Depends(get_db),
):
    job_id = uuid.uuid4()  # ← genera un ID nuevo, ignora el Idempotency-Key del cliente
    background_tasks.add_task(run_batch_recalculate, db, from_date, to_date, job_id)
    return BatchRecalculateResponse(jobId=job_id, status="QUEUED")
    # _headers["idempotency_key"] jamás se usa
```

**Remediación**:
```python
import asyncio
_running_jobs: dict[str, str] = {}  # idempotency_key → job_id (caché en memoria)

async def trigger_batch_recalculate(..., _headers: dict = Depends(...)):
    idempotency_key = _headers["idempotency_key"]

    # Deduplicar en memoria (válido para instancia única; usar Redis para escalado)
    if idempotency_key in _running_jobs:
        return BatchRecalculateResponse(
            jobId=UUID(_running_jobs[idempotency_key]),
            status="ALREADY_QUEUED"
        )

    job_id = uuid.uuid4()
    _running_jobs[idempotency_key] = str(job_id)
    background_tasks.add_task(_run_and_cleanup, idempotency_key, db, from_date, to_date, job_id)
    return BatchRecalculateResponse(jobId=job_id, status="QUEUED")
```

### [SEC-005] 🟡 MEDIO: Ausencia de headers de seguridad HTTP

**Archivo**: `app/main.py`
**Categoría**: A05 – Security Misconfiguration

**Descripción**:
La aplicación no configura ningún header de seguridad HTTP estándar. FastAPI/Starlette no los incluye por defecto. Los siguientes headers están ausentes en todas las respuestas:

| Header | Riesgo de ausencia |
|--------|--------------------|
| `X-Content-Type-Options: nosniff` | MIME sniffing por navegadores |
| `X-Frame-Options: DENY` | Clickjacking |
| `Strict-Transport-Security` | Downgrade a HTTP |
| `Cache-Control: no-store` | Caché de datos financieros en proxies |
| `Content-Security-Policy` | XSS (relevante si Swagger UI expuesto) |

El Swagger UI (`/docs`) y ReDoc (`/redoc`) están habilitados sin restricción de entorno, lo que expone documentación completa de la API a cualquier persona con acceso a la URL.

**Remediación**:
```python
# app/main.py
from starlette.middleware.httpsredirect import HTTPSRedirectMiddleware

@app.middleware("http")
async def add_security_headers(request: Request, call_next):
    response = await call_next(request)
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["Cache-Control"] = "no-store, no-cache"
    if settings.APP_ENV == "production":
        response.headers["Strict-Transport-Security"] = "max-age=31536000; includeSubDomains"
    return response

# Deshabilitar Swagger en producción:
app = FastAPI(
    docs_url="/docs" if settings.APP_ENV != "production" else None,
    redoc_url="/redoc" if settings.APP_ENV != "production" else None,
    ...
)
```

### [SEC-006] 🟡 MEDIO: Deserialización de Pub/Sub y Storage sin validación de tamaño

**Archivo**: `app/workers/pubsub_consumer.py:88`, `app/services/batch_service.py:48`
**Categoría**: A03 – Injection / DoS

**Descripción**:
Los mensajes recibidos desde Pub/Sub y los archivos descargados de Supabase Storage se deserializan directamente con `json.loads()` sin verificar el tamaño del payload antes del parsing. Un mensaje malicioso o un archivo corrupto de gran tamaño puede causar:
- Consumo excesivo de memoria (RAM exhaustion).
- Parsing lento de JSON muy anidado (ReDoS en jsonparsers).

Adicionalmente, en `batch_service.py`, si el bucket `event-logs` contiene archivos no-JSON (binarios, imágenes accidentales), la excepción `json.JSONDecodeError` se captura por el `except Exception` genérico y el job falla sin procesar los eventos válidos restantes.

**Evidencia**:
```python
# app/workers/pubsub_consumer.py:88 — sin límite de tamaño
raw = json.loads(message.data.decode("utf-8"))

# app/services/batch_service.py:47-48 — sin validación de tipo de archivo
raw = client.storage.from_(bucket).download(filename)
events = json.loads(raw)  # ← falla si el archivo no es JSON válido
```

**Remediación**:
```python
# Pub/Sub consumer: validar tamaño máximo
MAX_MESSAGE_BYTES = 10 * 1024 * 1024  # 10 MB (límite de Pub/Sub)
data = message.data
if len(data) > MAX_MESSAGE_BYTES:
    logger.warning("mensaje_demasiado_grande bytes=%d", len(data))
    message.nack()
    return
raw = json.loads(data.decode("utf-8"))

# Batch service: manejo granular de errores por archivo
for file_info in files:
    filename: str = file_info["name"]
    if not filename.endswith(".json"):
        logger.warning("archivo_no_json omitido filename=%s", filename)
        continue
    try:
        raw = client.storage.from_(bucket).download(filename)
        events = json.loads(raw)
    except (json.JSONDecodeError, UnicodeDecodeError) as e:
        logger.error("archivo_json_invalido filename=%s error=%s", filename, e)
        continue  # continuar con el siguiente archivo
```

### [SEC-008] 🔵 BAJO: `BackgroundTasks` instanciada como valor por defecto (anti-patrón FastAPI)

**Archivo**: `app/api/routes/batch.py:40`
**Categoría**: A05 – Security Misconfiguration (comportamiento inesperado)

**Descripción**:
`BackgroundTasks = BackgroundTasks()` como valor por defecto de parámetro es un **anti-patrón documentado en FastAPI**. Los valores por defecto mutables en Python se evalúan una sola vez al importar el módulo. Si FastAPI no inyecta la instancia correctamente en algún escenario (como tests o middlewares), la misma instancia podría ser compartida entre requests, causando condiciones de carrera donde las tareas de un request se ejecutan en el contexto de otro.

**Evidencia**:
```python
# app/api/routes/batch.py:40 — ❌ anti-patrón
async def trigger_batch_recalculate(
    background_tasks: BackgroundTasks = BackgroundTasks(),  # ← valor mutable por defecto
    ...
```

**Remediación**:
```python
# ✅ correcto — FastAPI inyecta BackgroundTasks automáticamente
async def trigger_batch_recalculate(
    background_tasks: BackgroundTasks,  # sin valor por defecto
    body: BatchRecalculateRequest | None = None,
    _headers: dict = Depends(require_headers_with_idempotency),
    db: AsyncSession = Depends(get_db),
):
```

### [SEC-009] 🔵 BAJO: Dockerfile ejecuta el proceso como root

**Archivo**: `Dockerfile`
**Categoría**: A05 – Security Misconfiguration

**Descripción**:
El `Dockerfile` no declara un usuario no-privilegiado. El proceso `uvicorn` corre como `root` (UID 0) dentro del contenedor. Si hay una vulnerabilidad de escape de contenedor o de path traversal en la aplicación, el atacante obtiene privilegios de root en el host.

**Evidencia**:
```dockerfile
FROM base AS runtime
COPY --from=builder /install /usr/local
COPY . .
EXPOSE 8070
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8070"]
# ← sin USER declarado → ejecuta como root
```

**Remediación**:
```dockerfile
FROM base AS runtime
COPY --from=builder /install /usr/local
COPY . .

# Crear usuario no privilegiado
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
USER appuser

EXPOSE 8070
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8070"]
```

### [SEC-010] 🔵 BAJO: Configuración de Settings acepta strings vacíos para credenciales críticas

**Archivo**: `app/config.py:25-34`
**Categoría**: A05 – Security Misconfiguration

**Descripción**:
Los campos `SUPABASE_URL`, `SUPABASE_ANON_KEY` y `DATABASE_URL` tienen como valor por defecto un string vacío `""`. Si el archivo `.env` no se configura correctamente, la aplicación arrancará sin error inmediato pero:
- El cliente SQLAlchemy intentará conectar a una URL vacía (error en el primer request).
- El cliente Supabase se inicializará con credenciales vacías y podría usar permisos públicos del proyecto.

No hay validación al startup de que las credenciales críticas estén presentes.

**Remediación**:
```python
from pydantic import field_validator, model_validator

class Settings(BaseSettings):
    SUPABASE_URL: str
    SUPABASE_ANON_KEY: str
    DATABASE_URL: str

    @model_validator(mode="after")
    def validate_critical_settings(self) -> "Settings":
        if not self.SUPABASE_URL or self.SUPABASE_URL == "https://xxxx.supabase.co":
            raise ValueError("SUPABASE_URL debe configurarse correctamente")
        if not self.DATABASE_URL or "user:pass@host" in self.DATABASE_URL:
            raise ValueError("DATABASE_URL debe configurarse con credenciales reales")
        if not self.SUPABASE_ANON_KEY or self.SUPABASE_ANON_KEY == "xxxx":
            raise ValueError("SUPABASE_ANON_KEY debe configurarse correctamente")
        return self
```

### [SEC-011] ℹ️ INFORMATIVO: Sin auditoría de dependencias en el pipeline de CI

**Archivo**: `.github/workflows/ci.yml`
**Categoría**: A06 – Vulnerable and Outdated Components

**Descripción**:
El pipeline de CI ejecuta lint y tests, pero no incluye ningún paso de auditoría de dependencias vulnerables. El proyecto usa librerías con historial de CVEs publicados (FastAPI, SQLAlchemy, google-cloud-pubsub, supabase). Una dependencia vulnerable puede introducirse silenciosamente en el futuro al actualizar `requirements.txt`.

**Recomendación**:
Agregar un paso de auditoría en `.github/workflows/ci.yml`:
```yaml
- name: Auditoría de dependencias
  run: |
    pip install pip-audit
    pip-audit --requirement requirements.txt --strict
```

O alternativamente integrar **Dependabot** en el repositorio (`.github/dependabot.yml`):
```yaml
version: 2
updates:
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 5
```
